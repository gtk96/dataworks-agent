# DataWorks Agent 通用化运行模型

> 目标：基础 Agent 能力做到线上通用，业务私有知识留在线下/本地知识库逐步沉淀。

## 1. 线上通用能力

以下能力应作为产品基础能力建设，不绑定某个业务域、家族、指标或目录：

- 正向建模：自然语言到 ODS/DWD/DIM/DWS 的 DDL、DML、FlowSpec、依赖和调度建议。
- 逆向建模：从存量表或节点读取结构、脚本、节点依赖和语义候选，生成可复用模型理解。
- 异常诊断：围绕任务、实例、节点、依赖、日志、执行底座做排查和恢复建议。
- 自主问数：在只读 SQL、口径澄清、资产证据和对账护栏下回答业务问题。
- 可视化执行：前端必须展示计划、执行步骤、产物、确认点、发布状态和下一步动作。

## 2. 本地/线下私有知识

以下内容不应硬编码到通用产品代码或通用前端示例：

- 具体业务家族、团队、人群、渠道、平台、店铺等私有词。
- 个性化指标口径、筛选条件、对账口径和业务解释。
- 本地探测得到的 DataWorks 目录、节点示例、表样例、专辑样例。
- 含账号、项目、AK/SK、Cookie、真实任务编号的敏感材料。

建议把这类内容沉淀在被 `.gitignore` 忽略的 `data/local_knowledge/`，只把结构模板、读取规则和安全边界放进版本库。

## 3. 执行与审批边界

当前确认的边界：

| 操作 | 边界 |
|---|---|
| Dev MaxCompute 建表 | 允许自动执行 |
| Dev DataWorks 建节点 | 允许自动执行 |
| 修改已有 dev 节点 | 执行前需要用户确认 |
| 删除节点 | 执行前需要用户确认 |
| 生产发布 | 永远人工确认，通过 Publish Gate |

前端、后端和文档都应使用同一套措辞，避免让用户误以为生产可自动上线。

## 4. DataWorks 目录判断规则

- 不把某个本地业务流程目录硬编码成通用默认体验。
- 创建 DataWorks 节点时优先按 FlowSpec 的 `script.path` 落位；新版数据开发使用 `scene=DATAWORKS_PROJECT`，`container_id=None`。
- Hologres 节点必须落在真实存在的 Hologres 目录树；目录是否存在应通过配置、OpenAPI/Cookie/DataStudio 树或本地知识库判断。
- 本地发现的目录样本属于私有知识，应写入 `data/local_knowledge/` 或用户指定知识库，不进入通用前端提示。

## 5. 验收优先级

用户关注的是可视化 Agent 的真实体验，因此每次改造至少要覆盖：

1. 后端最小语法/目标回归，确保核心 API 不坏。
2. 前端 `npm run build` 通过。
3. 真实 HTTP/浏览器烟测，确认页面能打开、关键文案和交互可见。

单测用于守住逻辑回归，但不能替代前端真实体验验收。
